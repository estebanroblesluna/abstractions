  Javascript
  (por Leandro Yalet) - 03/Abril/2008 - Permalink
 

          JavaScript fue originalmente desarrollado por Brendan Eich para Netscape bajo el nombre Mocha, luego llamado LiveScript y finalmente renombrado JavaScript. El cambio de nombre de LiveScript a JavaScript coincide aproximadamente con el soporte de Netscape para las tecnologías Java y fue incorporado en 1995 en el Netscape Navigaror 2.0 beta.

      Como noticia de último momento en Diciembre AOL (que compró a Netscape hace unos años) fijó la fecha del entierro para el Netscape Navigator, a partir del 1 de febrero de este año dejó de dar soporte, con lo cual la cuna de JavaScript dejó de existir. Para más información en este link hay una línea temporal de navegadores web.

      Para continuar con el pie correcto, se debe aclarar que JavaScript != Java (para los smalltalkers JavaScript ~= Java), simplemente se tomó la popularidad que tenía Java en sus comienzos. Las aplicaciones Java son independientes de una página web y compiladas en bytecodes, mientras que los programas JavaScript están embebidos en una página web y deben correr en un browser.

      La idea original fue hacer la programación más accesible que Java y de algún modo hacer que el navegador de Netscape sea más abierto para quienes no programaran en ese lenguaje.

      En cuanto a la sintaxis JavaScript es similar a C, Perl y Java. Por ejemplo las sentencias if, while y los loops for son idénticas. Al igual que Perl, es un lenguaje orientado a objetos, interpretado y no compilado.

      A raíz de que JavaScript está asociado con un navegador, está altamente integrado con HTML. Mientras HTML es manejado por la propia biblioteca de red y de renderizador de gráficos, los scripts de JavaScript pueden ser ejecutados por un intérprete normalmente incorporado al navegador. Cuando el navegador solicita una página, el servidor envía el contenido total de un documento, incluyendo HTML y sentencias JavaScript, a través de la red hacia el cliente. Cuando la página carga, el contenido HTML es leído y renderizado línea por línea hasta la apertura de un tag de JavaScript, a partir de ese momento el intérprete de JavaScript toma el control. Cuando el tag de cierre es alcanzado, el proceso del HTML continúa.

      Cuando hablamos de un intérprete de JavaScript en un navegador se trata de client-side JavaScript (CSJS). A pesar de que JavaScript es altamente utilizado como un lenguaje client-side y embebido en un navegador web, puede ser usado en contextos distintos de un navegador, por ejemplo Adobe Acrobat y Adobe Reader. Netscape creó un server-side JavaScript (SSJS) llamado LiveWire para que sea programado como un lenguaje CGI como Perl o ASP.

      El motor más conocido es el SpiderMonkey (code name adquirido en Netscape) que fue convertido en open source. Ejemplo de aplicaciones que utilicen este motor es el tan conocido navegador Mozilla Firefox y Yahoo Widgets. Pero SpiderMonkey, desarrollado en C, tiene un hermano llamado Rhino, implementado completamente en Java. Rhino es utilizado en los servidores Helma y Phobos. Rhino es parte de la JDK 6.

Qué no es JavaScript

      JavaScript no es HTML, pero un script puede estar embebido en un documento HTML y contenido dentro de tags HTML. Javascript no comprende HTML pero puede contenerlo dentro de sus sentencias.

      JavaScript está basado en objetos pero no es estrictamente orientado a objetos porque no soporta el tradicional mecanismo de herencia y las clases encontradas en lenguajes orientados a objetos como Smalltalk, Java o C++. Palabras reservadas como private, protected y public no son aplicables a los métodos de JavaScript como en Java y C++.

      JavaScript no es el único lenguaje que puede ser embebido en una página web. VBScript por ejemplo, desarrollado por Microsoft, es parecido a JavaScript pero funciona únicamente en Internet Explorer.

Los usos de JavaScript

      Como diría una conocida lifiana “¿Pa’ que sirve?”[1], los programas JavaScript son usados para detectar y reaccionar a los eventos iniciados por el usuario, como el movimiento del mouse sobre un link o una imagen. Estos pueden mejorar una página con ayuda en la navegación, mostrando mensajes, cajas de diálogo, intercambio de imágenes (algo que todavía se utiliza), carritos de compra y así sucesivamente. JavaScript da el control a la apariencia de las páginas desde el comienzo del parseo del documento. Sin ninguna comunicación con el servidor es posible validar lo que el usuario ingresó en un formulario antes de que lo envíe. Se puede verificar qué plug-ins posee el usuario y enviarlo a otra página para que obtenga los plug-ins que necesite. Posee funciones para strings y expresiones regulares para validar direcciones de mail, números de teléfono, etcétera. Es usado para cálculos aritméticos, manejos de fechas y tiempo, y trabaja con arreglos, strings y objetos. Además puede leer y escribir cookies y crear dinámicamente HTML.

Una historia con muchos problemas
     
      Los programadores web encontraron en JavaScript una programación divertida. Hasta cierto punto, eso significó que muchos programadores no lo tomaran en serio. Originalmente un sufrimiento para muchos, comenzó siendo aceptado por cientos de programadores. Como los navegadores se hicieron más poderosos y podían ejecutar scripts más rápido, los programadores descubrieron que Javascript era, en verdad, una herramienta viable de desarrollo. Y todo tomaba unas pocas líneas de código para hacer que funcionen las cosas.

      JavaScript comenzaba a hacer olas y Microsoft, el competidor de Netscape, no pudo ignorar la situación y decidió crear su propia versión de JavaScript a la cual llamó Jscript.

      La primera versión oficial de JScript fue liberada en Julio de 1996, en el Internet Explorer 3.0. A partir de entonces había JavaScript y JScript, y el resultado de esta división del lenguaje de Netscape y Microsoft tuvo repercusiones que tienen eco hoy en día. El original JScript era una versión mucho más reducida que JavaScript 1.1 y las diferencias confundían a los programadores. En ese punto, Microsoft creó su propio lenguaje de script, VBScript, basado en Visual Basic, y tuvo la esperanza de ser un próspero rival de JavaScript, pero este último fue el ganador.

      Microsoft con JScript, para traer un completo soporte para JavaScript 1.1, relanzó una segunda versión de Internet Explorer 3.0. Entonces comenzaron los problemas de cross-browser y versiones cross-browser en los scripts, problemas que hacen difícil la vida de los programadores de JavaScript desde entonces. Los programadores descubrieron que aunque JScript se parecía a JavaScript, algunos scripts corrían en Netscape y no en Internet Explorer y viceversa.

      Con la esperanza de detener el caos, Netscape y Sun buscaron una tercera parte que estandarizara JavaScript, y se dirigieron hacia la European Computer Manufacturers Association (ECMA, http://www.ecma.ch).

      Desafortunadamente para los programadores, la estandarización, que comenzó a fines de 1996, tomó mucho tiempo, y mucho más tiempo en ser adoptado. Uno hubiese esperado que el lenguaje estandarizado se llamase JavaScript o JScript, pero en vez de eso se llamó ECMAScript. Entonces actualmente tenemos tres estándares: JavaScript, JScript y ECMAScript.

      A pesar de que ECMAScript tuvo cierta cantidad coherente de aciertos, ambos navegadores fueron por sus propios caminos. El resultado es que Internet Explorer permitió dos maneras para hacer las cosas (JScript y ECMAScript) y Netscape lo hizo a la manera de JavaScript y ECMAScript. Un gran problema fue que ECMAScript tenía estandarizado el núcleo de JavaScript en sí mismo, pero no todos los recursos estaban disponibles en los navegadores.

      Estos recursos están constituidos por los “browser objects”. Cada navegador tenía su propio conjunto de objetos, como el objeto document, que permite que uno trabaje con la página web y el objeto window que permite trabajar con el browser en sí mismo. Una gran cantidad de la programación de JavaScript es sobre los lineamientos de estos objetos.Estos son enteramente diferentes en los dos navegadores, es fácil ver que el mismo script tenga problemas en uno o el resto de los navegadores (por eso mismo muchos hoy en día siguen programando diferentes scripts para la misma página web para los distintos navegadores).

      El caos fue puesto a la vista por los diferentes conjuntos de objetos de los navegadores amenazando a JavaScript y JScript, entonces Microsoft, Netscape y muchas otras corporaciones de la World Wide Web Consortium (W3C, http://www.w3.org) aportaron coherencia a estos objetos para poder construir un JavaScript accesible.

      El conjunto original de objetos, antes de que la W3C comenzara su trabajo, llegó a ser conocido como Document Object Model (DOM) "level 0". El DOM level 1, el cual llegó a ser una especificación de la W3C oficial a fines de 1998, estandarizó muchos aspectos de los objetos encontrados en los navegadores, pero dejó afuera a muchos otros. DOM level 2 fue creado como especificación en Noviembre de 2000 y completó mucho más detalles. DOM level 3 introduce, entre otras cosas, Xpath.

La curva de la evolución

      El mundo JavaScript hizo un gran avance desde su edad de piedra con pequeños scripts que llenaban la pantalla de “alerts”, banners, menues desplegables, imágenes que se intercambiaban al pasar el mouse por encima, etcétera; a este tipo de cosas se las conoce como DHTML: Dynamic HMTL que consiste en la combinación de HTML 4.0, el Document Object Model (DOM), JavaScript y CSS (Cascading Style Sheets). Desde el estilo inicial que contenga una página con CSS podemos comenzar utilizando el DOM y JavaScript para cambiar dinámicamente el estilo de la página luego de que fué cargada.

      La utilización de frames en las páginas para cargar otras más pequeñas necesitó de JavaScript para establecer una comunicación entre las mismas, este tipo de técnica era una manera primitiva de lo que conocemos como Ajax, este término fue acuñado recientemente en el 2005. Con la llegada del objeto XMLHttpRequest comenzaron a aparecer pequeñas bibliotecas para utilizar dicha técnica.

      Actualmente y a raíz de la denominada Web 2.0, que todo el mundo conoce pero que abarca muchos conceptos, podemos encontrar bibliotecas que permiten portabilidad y claridad en el código entre otras cosas, por fomentar el uso de una notación orientada a objetos.

      Lo que menos uno se imaginó es que al utilizar cualquiera de estas herramientas el trabajo se vería reducido en muchas líneas de código y cuando se habla de muchas, cabe aclarar que este tipo de bibliotecas se encarga de parchear aquellas particularidades de cada browser creando de esta manera scripts cross-browser. Una maravillosa consecuencia es la reinvención de la rueda[2] pero versión Web 2.0: por ejemplo menues desplegables pero animados con efectos y configurables. Pero más allá de lo que se pueda decir, la comunidad que se encarga de utilizar y crear estas bibliotecas, se preocupa por cumplir con los estándares y hoy en día se puede encontrar una extensa documentación del uso de sus APIs. En las primeras épocas era muy común realizar copy-paste de lo que encontraba, lo mejoraba y verificaba su funcionamiento, pero la nueva costumbre es utilizar distintos bibliotecas e incorporar distintos accesorios para armar un nuevo script con mucha más funcionalidad y compartirlo con la comunidad.

      En esta línea de tiempo sobre selectores de CSS[3] se puede ver la explosión de estas bibliotecas
entre los años 2006 y 2007. El 2008 deparará muchas más sorpresas y tal vez en octubre esté lista la versión 2 de JavaScript (hoy en día la mayoría de los navegadores implementar la versión 1.7) que traerá consigo clases, interfaces, paquetes y muchas mejoras.
