  Struts 1 vs struts 2
  (por Federico Carpi) - 03/Enero/2008 - Permalink
        ¿Que es Struts 1-2?    

   Todos los que estamos en el área de desarrollo de software, más puntualmente en el área de desarrollo web, conocemos el tan afamado Framework Struts.

      Básicamente Struts es una herramienta de soporte para el desarrollo de aplicaciones Web bajo el patrón MVC en la plataforma J2EE. Una de las características principales es que permite reducir el tiempo de desarrollo. Su carácter de "software libre" y su compatibilidad con todas las plataformas en que Java esta disponible, lo convierte en una herramienta altamente disponible.

      A fines del 2006 surge la evolución de Struts 1 el nuevo Framework Struts 2 basado completamente en el Framework WebWork 2. Luego de que las comunidades de ambos proyectos trabajarán independientemente, unieron fuerzas y de esta manera nació Struts 2.

      El objetivo del Struts 2 es el mismo que el de Struts 1, brindar soporte para el desarrollo de aplicaciones Web bajo el patrón MVC. La filosofía de trabajo se basa en 3 conceptos “construir” (build), “desplegar” (deploy) y “mantener” (maintain).

¿Qué similitudes poseen las dos versiones?    

   Ambas versiones de struts proveen tres componentes claves.
1)Un “request handler” que mapea clases Java a URIs de aplicaciones Web.
2)Un “response handler” que mapea nombres lógicos a páginas del servidor o otros recursos web.
3)Un conjunto de tags que permiten crear aplicaciones robustas basadas en formularios.       En Struts 2 las tres componentes han sido rediseñadas y mejoradas pero las características de la arquitectura sigue siendo la misma.

¿Qué cosas cambiaron en Struts 2?  

 - Diseño mejorado: Todas las clases de Struts 2 están basadas en interfaces. Las interfaces del núcleo son independientes de la arquitectura HTTP.

 - Valores por defecto: La mayoría de los elementos de configuración poseen un valor por defecto que puede ser seteado (y no hay que preocuparse por configurarlo nuevamente).

 - Resultados mejorados: A diferencia de los antiguos “ActionForward” de Struts 1, los resultados de Struts 2 son tipificados permitiendo preparar el response en los casos que sea necesario.

 - Tags mejorados: Con los nuevos tags de struts 2 es posible crear páginas consistentes con menor escritura de código.

 - Soporte para Ajax: El “theme” ajax permite a las aplicaciones interactuar con el servidor mediante requerimientos asincrónicos.

 - Inicio rápido: Muchos cambios pueden ser realizados dinámicamente (“on the fly”) sin ser necesario el reinicio del contenedor web.

 - Test de las Acciones: Las acciones de Struts 2 son independiente de la arquitectura HTTP con lo cual pueden ser fácilmente testeadas mediante test de unidad sin la necesidad de recurrir a objetos que simulen el comportamiento (“mock objects”) de los objetos reales.

 - Integración con Spring: Struts 2 es “Spring-aware”. En otras palabras las acciones pueden ser creadas como beans de Spring.

 - Plugins: Las extensiones de Struts 2 pueden ser agregadas simplemente copiando un “jar”. No es necesaria una configuración manual.

 - Formularios: En Struts 2 no existe más el concepto de formulario tal como existía en Struts 1. Ahora es posible usar cualquier JavaBeans contenido en una acción o escribir directamente sobre los atributos de una acción. Por otro lado, ¡Ya no es necesario que todas las propiedades sean sólo String! Struts 2 convierte automáticamente los valores cargados en el formulario web a los tipos de valores que posea el JavaBean referenciado.

 - Acciones POJO: Cualquier clase puede ser usada como una acción. Ya no es necesario implementar una interface o subclasificar alguna clase del framework.

¿Es difícil migrar de Struts 1 a Struts 2?

      Solo un poco. El trabajo en sí mismo no es tan complicado, pero la suma de esfuerzos no es trivial. Migrar las páginas y las acciones puede tomar cierto tiempo y esfuerzo con un costo considerable. Para aplicaciones existentes que prácticamente no serán modificadas o que, al contrario, serán “altamente” modificadas la migración no valdrá la pena en función de los costos/beneficios y el esfuerzo que requerirá.

¿Struts 1 quedó obsoleto?

      Definitivamente no. Existe una gran comunidad de desarrolladores que están usando Struts 1 en producción, cientos de equipos continuarán basando sus desarrollos en Struts 1, y obviamente es y será necesario proveer soporte para proyectos existentes por muchos años más.

      De todas formas, si se está iniciando un nuevo proyecto y es posible decidir entre ambas versiones, es un momento excelente para considerar si continuar usando Struts 1 o probar Struts 2.
