  ¿Qué es integración continua (IC)?
  (por Germán Insogna) - 13/Marzo/2008 - Permalink
 

      La Integración Continua es una práctica de desarrollo de software donde los miembros de un equipo de desarrollo integran su trabajo con frecuencia, por lo general se hace una integración una vez al día, teniendo múltiples integraciones diarias por equipo. Cada integración es verificada y posiblemente realizada por un sistema automatizado de construcción (incluídas las pruebas). De esta manera se pueden detectar errores en la integración lo más rápido posible. Los equipos de desarrollo encuentran en este enfoque una solución que los lleva a reducir los problemas de integración y permitir el desarrollo de software rápidamente.

¿Por qué usar IC?

     La Integración Continua es una metodología de detección de fallos de manera temprana. Por ende, no corrige errores, pero sí los hace más fáciles de encontrar y eliminar. Cuanto mas rápido se detecte un bug, más fácil será corregirlo. Solucionar un problema después de dos semanas de trabajar de forma independiente, es mucho más costoso que corregirlo después de dos horas de trabajo (por lo que podemos decir que es mucho más manejable).

      Al tener integración continua eliminamos uno de los mayores obstáculos para un frecuente deploy. Realizar un deploy con mayor frecuencia permite a sus usuarios obtener nuevas características con mayor rapidez, para dar más rápido feedback y tener más colaboración en el ciclo de desarrollo. Esto ayuda a romper las barreras entre los clientes y los desarrolladores.

¿Qué se necesita para hacer IC?

• Un repositorio de código (Subversion, CVS)

• Automatización de los procesos de build: La comunidad Java desarrollo Ant.

• Test automáticos: Una buena práctica para descubrir bugs de forma temprana y de forma eficiente es incluír test automáticos en el proceso de build.

• Diariamente realizar commits: La integración es principalmente comunicación, permite a otros desarrolladores contar con los últimos cambios que fueron hechos. Una de las precondiciones para un desarrollador es realizar commits de sus cambios en la línea base de desarrollo (mainline).

• Build en una máquina dedicada a la integración: realizando commits diariamente, se tiene que garantizar que la línea base quede en un estado óptimo. En la práctica esto no ocurre. Generalmente no se hacen las actualizaciones de las últimas versiones del repositorio antes de realizar un build privado, para luego subir los cambios. Otra causa es la diferencia de ambientes entre los desarrolladores.
Un servidor de integración continua actúa como monitor de los cambios que se realizan en el repositorio al subir una nueva versión. De esta manera se puede notificar a los desarrolladores si el build fue exitoso (generalmente con un mail).

• Builds rápidos: Uno de los objetivos de la IC es obtener un feedback rápido. Hay que tratar que este tiempo de build sea el menor posible.

• Test en un ambiente clonado de producción: Testear en un entorno lo más parecido posible al entorno de producción. Si las pruebas se realizan en un entorno distinto, cada diferencia que existe se convierte en un riesgo.

• Tener disponible el último build: una de las partes más difíciles del desarrollo de software es asegurarse que se está construyendo lo que uno quiere. Es muy difícil especificar y además que esta sea correcta. Resulta más fácil ver algo que no está del todo bien y decir que se necesita cambiar. El proceso de desarrollo ágil explota esta característica. Para esto es necesario tener el último ejecutable disponible con las últimas versiones para demostraciones, pruebas o simplemente para ver lo que se avanzo en la semana.

• Status broadcasting: IC se basa todo en la comunicación, por lo que se desea que todos en el proyecto puedan ver el estado del sistema y los cambios que se realizaron.

• Automatización del deploy: para realizar IC se necesitan múltiples ambientes para poder probar, esto debe ser de forma automática. Es importante tener una secuencia de comandos que permita realizar el deploy de la aplicación fácilmente en cualquier entorno.

Sabores en que viene la IC
Tipos de Build

     Hay tres tipos de Build: un desarrollador puede realizar un build local, un build que integra los resultados con el resto del equipo o un build para realizar el deploy del software para el usuario.

Build locales

     Un desarrollador corre una construcción privada antes de subir los cambios al repositorio. Realizando esta operación nos aseguramos de integrar nuestros cambios con los últimos cambios disponibles en el repositorio. Esto puede prever que subamos cambios de código que no compile.

Pasos para realizar un build privado:
1.Check out de las últimas versiones del repositorio.
2.Realizar nuestros cambios al código.
3.Traer los últimos cambios del repositorio, para realizar un merge si es necesario con nuestro nuevo código.
4.Correr un build que incluya la ejecución de todos los test de unidad.
5.Hacer commit del código al repositorio.

Builds de integración: Integra los cambios realizados en el repositorio para el resto del equipo en la línea base (head o trunk). Idealmente esta integración debería realizarse en una máquina dedicada.

Integración de despliegue (deploy): La integración de despliegue prepara la aplicación para los usuarios. Uno de los logros de la IC es crear soft deployable. Una entrega a los usuarios ocurre al final de una iteración o en algún otro hito. Se deben incluir los test de cargas y performance además de los test de aceptación. Esta integración también suele usarse para comenzar el proceso de QA si es que hay un equipo separado.

Mecanismos de Construcción

No todas las construcciones se realizan de la misma forma. Para realizar una construcción de la mejor manera, se debe considerar el propósito y la frecuencia. En algunas situaciones se deben correr demasiados scripts o deben respetar ciertas dependencias que no permiten que se corran automáticamente. En otros casos, se pueden correr bajo IC. La siguiente lista describe los tipos de mecanismos de construcción.

A demanda:: es un proceso realizado por el usuario, en el cual el usuario inicia manualmente la integración.

Programado: un proceso programado iniciado por tiempo, sin importar si ha ocurrido algún cambio. Se puede correr en un horario no laboral para realizar pruebas exhaustivas. Se puede hacer usando un cron. Muchos de los servidores de IC soportan tareas programadas.

Poll de cambios: un proceso se despierta en intervalos regulares y busca cambios de versiones en el repositorio. Si son detectados cambios, se corre el proceso de integración.

Cambio de eventos: es como un pool de cambios, con la diferencia en el momento que se ejecuta la integración. En el momento que suben una nueva versión al repositorio se dispara el proceso de integración.
